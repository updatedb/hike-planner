# Hike Planner SOP — 徒步行程规划助手

**版本**: v3.1
**创建日期**: 2026-05-14
**最后更新**: 2026-05-27（v3.2 裁剪：移除订票/购买/候补/日历/邮件/记忆/主动推送等 scope creep，对齐只查不买+用户驱动原则）

## 一、职责定位

**角色**: 徒步行程规划助手

**核心职责**:
- ✅ **行程前** — 根据用户明确指令规划徒步路线，查询可用的票务、酒店信息（只查不买），输出行程计划到本地文件
- ✅ **行程中** — 根据用户提供的短信/订单信息更新行程状态，记录实际出行日志
- ✅ **行程后** — 汇总实际行程数据，归档到 completed 目录

> ⚠️ **文件写入规则**：
> - `hike-init` 首次创建行程 → 弹窗确认 outputDir 后，写入状态文件 + 行程计划文件
> - `hike-log` 含疑似短信内容 → 弹窗确认后才解析并保存
> - `hike-add / hike-del / hike-reorder` 等编辑命令 → 用户主动操作，直接覆写行程计划文件（不弹窗，等同于"保存"）
>   - ⚠️ `hike-del` 删除日程后不可撤销，建议执行前手动备份行程计划文件
>   - ⚠️ `hike-reorder` 重排后原顺序不可恢复
> - `hike-select output` 切换输出目录 → 用户主动命令，直接写入 `~/.hike-planner/config.json`（不弹窗）
> - 所有持久化文件清单：`~/.hike-planner/config.json`（全局配置）、`{outputDir}/.hike-planner-state.json`（行程状态）、`{outputDir}/upcoming/{tripId}/{planFilename}`（行程计划）

## 二、Skills 清单

| Skill | 用途 | 触发场景 |
|-------|------|---------|
| hike-planner 内置 | 徒步路线搜索+行程地图渲染+GPX/KML轨迹解析 | "搜索古蜀道路线"、"生成行程地图"、"导入GPX/KML轨迹" |
| `12306-train-assistant` | 火车票时刻/票价/经停站查询 | "查火车票"、"D2008经停站"（**强制使用**） |
| `amap-lbs-skill` | 距离计算、路线规划、地理编码 | "计算两地距离"、"校验路线"（**强制使用**） |
| 'flyai','飞艾','飞猪' | 机票/酒店查询比价 | 「搜剑门附近酒店」「搜北京到成都最便宜的机票」 |
| `weather` | 天气预报 | "徒步那天天气如何" |
| `web_search` / `tavily_search` | 实时信息查询 | "五一川西路况" |

> ⚠️ **数据源分工规则**:
> - **火车票** → 必须用 `12306-train-assistant`（flyai 的火车数据不如12306准确，曾出现直达车误显示为需换乘的错误）
> - **机票/酒店/景点** → `flyai`（飞猪MCP）
> - **距离/路线** → `amap-lbs-skill`（强制，禁止用经纬度公式估算）
> - **目的地知识** → Wikipedia + 小红书（维基百科为主，小红书补充真实用户体验）
> - **两步路/小红书均不可用时** → 明确告知用户信息源不可用，请用户人工提供路线信息或确认搜索结果的可靠性
> - 禁止用非官方渠道查询火车票余票/经停站
>
> ⚠️ **只查不买**：所有票务和酒店信息查询结果仅为规划参考。不执行订票、下单、候补、支付操作。
>
> ⚠️ **先检索，再撰写**：禁止捏造信息。Wikipedia 用于地理/历史/人物事实，小红书用于真实用户体验/补给点/当季路况，12306 用于火车票，flyai 用于机票/酒店，amap-lbs-skill 用于距离/路线。

## 三、完整工作流

### 3.1 行程规划流程（出发前）

```
收集需求 → 锁定大交通 → 起点顺排 → 终点倒排 → 安排小交通 → 填补日常安排
```

**第一步：收集需求（必须先做）**

> ⚠️ 没有需求就没有规划，需求不清不进入下一步

| 信息项 | 说明 | 示例 |
|--------|------|------|
| 出行日期 | 出发日期+返回日期（单程可不填返回）| "5月13日去，5月16日回" |
| 出发地 | 出发城市/地点 | "北京"、"成都" |
| 目的地 | 主要目的地 | "剑门关"、"古蜀道" |
| 返程地 | 单程可不填 | "回北京" |
| 人员数量 | 出行人数 | "2人"、"1大1小" |
| 偏好 | 交通方式/住宿/餐饮/体力分配 | "自驾"、"少走路"、"住亚朵" |

**步骤说明**:

1. **收集需求（先问清楚）** — 日期、地点、人数、偏好，不清楚时必须询问用户，不能猜测
2. **搜索路线** — 徒步轨迹搜索为 hike-planner 内置能力。两步路为主：访问 `https://www.2bulu.com/track` 搜索轨迹（如 web_fetch 无法访问，使用 browser 工具以真实浏览器打开该链接）；小红书/B站/YouTube 补充图片和视频。如有用户提供的 GPX/KML 文件，优先解析获取精确坐标和实测距离。

    **小红书搜索策略（强约束）**：
    
    a. **关键词组合模板**（每次必须覆盖）：
    
    | 类别 | 搜索关键词 | 目的 |
    |------|-----------|------|
    | 避坑 | `<目的地> 避坑` | 提前发现常见陷阱 |
    | 时效攻略 | `<目的地> 最新攻略 <年份>` | 获取最新路线/路况 |
    | 天气 | `<目的地> <月份> 天气` | 季节穿衣/装备参考 |
    | 防宰 | `<目的地> 防宰` | 物价/餐饮/交通防坑 |
    | 物价 | `<目的地> 物价` | 预算评估 |
    | 景点评估 | `<景点名> 值不值得去` | 是否纳入行程 |
    
    b. **陷阱识别规则**：
    - 90% "精品一日游" 帖子是商业推广 → 看评论区 + 发帖时间筛真实内容
    - 帖子含 "私信/加微信/进群" → 直接标记为推广，降权
    - 同一账号批量发同类帖 → 推广号
    - 连续多个帖子字数高度相似 → 疑似AI生成，降权
    - 仅配航拍/宣传图无手机随拍 → 商业文案
    
    c. **评论区必看**（评论区比正文更真实）：
    - 负面评论 "踩坑了/不值得" → 重点分析原因
    - 追问回复 "求路线/怎么去" → 实用信息源
    - 时间类信息 "排队XX小时" → 安排 buffer
    - "推荐XX师傅/司机" → 可能真实，需交叉验证
    
    d. **可信度评级标准**：
    
    | 信号 | 加分 | 减分 |
    |------|------|------|
    | 发帖时间 ≤ 3 个月 | +2 | — |
    | 手机随拍 + 路线描述 | +2 | — |
    | 花费明细 | +1 | — |
    | 有槽点/负面评价 | +1 | — |
    | 含 "私信/加微信" | — | -3 |
    | 纯航拍/宣传图 | — | -2 |
    | 账号无历史内容 | — | -2 |
    | 全文只夸不批 | — | -1 |

3. **接收并解析轨迹文件（如有）** — 如用户发送了 GPX/KML 轨迹文件（Garmin Connect/佳明/两步路导出/Google Earth等），解析并提取：

    **必须提取的数据**：
    - 起终点 GPS 坐标（lat/lng）— 用于地图渲染，直接定位，无需地理编码
    - 关键节点的 GPS 坐标列表（waypoints）— 存入 `hikingRoutes[].waypoints`，地图渲染时优先使用
    - 总距离、累计爬升/下降、海拔曲线

    > ⚠️ **GPX/KML 为最高优先级数据源**：当网络搜索（两步路/小红书等）的路线数据与上传的 GPX/KML 轨迹存在偏差时，**以 GPX/KML 为准**。轨迹文件是实地记录的实测数据，距离、爬升、节点坐标均以轨迹文件数据覆盖搜索结果中的估算值。
    >
    > 🛰️ **GPS 坐标直接用于地图渲染**：从 GPX/KML 提取的 waypoint 坐标（lat/lng）无需经过高德地理编码，直接传给 `render-itinerary-map.js --coords` 生成精确路线图。只有没有 GPX/KML 时才走地理编码流程。
    >
    > GPX/KML 轨迹 ≠ 实际古道路线：GPS 记录的是用户走过的实际路径，可能绕行或偏移。但即便如此，轨迹数据仍是可获得的最高精度数据源。

4. **锁定大交通（核心约束）** — 飞机、高铁等大交通时间是固定锚点，一切围绕它展开
5. **起点顺排** — 从出发地开始，按时间顺序安排每段小交通+活动
6. **终点倒排** — 从大交通终点回推，确保每个节点时间衔接合理
7. **安排小交通** — 城际交通、包车、公交等基于前两步的时间窗口安排
8. **填补日常安排** — 餐饮、休息、景点等填入时间空隙
9. **生成行程地图** — 行程确认后，使用 `hike-planner/scripts/render-itinerary-map.sh` 为每日行程生成高德地图可视化链接。

    **地理编码消歧义（重要）**：同一地名可能存在于多个省份（如"姜庄子村"），地理编码前必须附加省市区县前缀。建议传入 `destinationRegion`（如 "北京市延庆区"）作为定位范围，消除重名歧义，确保路线计算正确。

    调用时传入节点对应的省市区县前缀，例如：
    ```bash
    # 格式：节点列表 | 城市 | 路线类型 | 省市区县前缀列表
    bash render-itinerary-map.sh "姜庄子村|小海坨|大海陀村" "延庆" "walking,walking" "北京市延庆区,北京市延庆区,河北省赤城县"
    ```

10. **模板完整性检查** — 行程计划文件生成完成后，调用 `hike-planner.js` 的 `validatePlan(content, trip)` 函数对照 PLAN_TEMPLATE 检查所有板块是否完整。检查项包括：6 个顶级章节（总览/每日安排/行程详情/徒步路线详情/装备清单/待办事项）、文化分类 ≥3 个、每日地图链接、装备清单表格、待办事项等。如有缺失或警告，立即补齐后重新验证。

    **导航类型规范**：
    - 🥾 徒步段 → `walking`
    - 🚗 出租/包车/自驾 → `driving`
    - 🚌 公交段 → `driving`
    - 🚄 火车 / ✈️ 飞机 → 直线连接（两点间用 straight line）

### 3.2 时间规划核心原则

> ⚠️ **大交通是锚点，是约束，是起点也是终点**

**正确顺序**：
1. 确认大交通时间（机票/火车票）
2. 以大交通为锚点，**倒排**当天返程活动
3. 以大交通为锚点，**顺排**去程活动
4. 检查各节点时间衔接是否满足最小候车/候机时间

**候车/候机时间规范**：
- 高铁站：候车不超 30 分钟为佳
- 机场：提前 1 小时到达办理登机

### 3.3 行程执行流程（出发后）

```
用户查询状态 → 查看今日计划 → 记录实际进度 → 突发应变 → 行程收尾
```

> 所有查询均为用户驱动。用户可通过 `hike-status`（概览）、`hike-today`（今日时间线+地图）、`hike-log`（记录实际信息）主动获取和更新行程状态，Skill 不进行未经请求的主动推送。

### 3.4 行程收尾流程（返回后）

```
记录实际出行数据 → 与计划对比 → 文件归档到 completed 目录
```

## 四、启动检查清单

每次行程规划启动前：

- [ ] 理解用户出行目的（徒步/出差/家人游/个人探索）
- [ ] 确认时间范围、交通偏好、住宿偏好（§3.1 第一步）
- [ ] 创建行程目录前**征得用户明确同意**（告知输出路径和数据范围）

- [ ] 查天气确认是否合适
- [ ] 火车票用 12306 skill 验证（§七）
- [ ] 城际/包车段距离用 amap-lbs-skill 校验（§八）
- [ ] **向用户说明高德地图 API 调用会传输节点名称**
- [ ] 生成行程计划文件并**用户确认后**才写入磁盘

## 五、行程规划标准文档结构

### 5.1 文件路径

- 即将出行：`{outputDir}/upcoming/<destination-slug>-<YYYY-MM>/<实际标题>.md`
- 已完成：`{outputDir}/completed/<destination-slug>-<YYYY-MM>/<实际标题>.md`

**命名规则**：

| 组成部分 | 规则 | 示例 |
|---------|------|------|
| 目录名 | `<destination-slug>-<YYYY-MM>`（pinyin/英文小写 + 连字符，日期带分隔符） | `haituo-2026-06`、`japan-tokyo-kyoto-osaka-2026-08` |
| 文件名 | `<实际标题>.md`（文档一级标题，去 emoji，保留 `·` `→` 等合法字符） | `海坨山姜庄子村小环线出行计划.md`、`日本·东京→京都→大阪 8天行程规划.md` |

**示例**：

| 目录 | 计划文件 |
|------|---------|
| `haituo-2026-06/` | `海坨山姜庄子村小环线出行计划.md` |
| `wutaishan-2026-06/` | `五台山出行计划.md` |
| `japan-tokyo-kyoto-osaka-2026-08/` | `日本·东京→京都→大阪 8天行程规划.md` |

### 5.2 文档模板

> 完整模板见 `PLAN_TEMPLATE.md`。以下为核心结构概要，详细内容和所有占位符说明以模板文件为准。

| 章节 | 说明 |
|------|------|
| 总览 | 日期/星期/行程概要/出行方式/班次/酒店/含早/天气 汇总表 |
| 每日安排 | DAY N 时间线表（时间/区间/节点详情/费用/备注） + 地图链接 + 本日费用 |
| 行程详情 | 目的地文化分类 3-5 个（地理风貌/历史渊源/人文诗词/遗存遗迹/美食特产等）。**多日行程按日扩展**，每重要活动日独立一节 |
| 徒步路线详情 | 距离/爬升/下降/预计用时/路线类型/路线分类/关键节点/轨迹来源（GPX/KML）/轨迹地图/提示 |
| 装备清单 | 按类型分类（鞋/衣/包/导航/水/食物/防晒/药品/证件/电子/其他） |
| 待办事项 | 订票/订酒店/装备/保险/离线地图 |

### 5.3 模板使用规范

| 规范 | 说明 |
|------|------|
| 同一酒店多日入住 | 不需要每天退房+再入住，行李寄存前台即可，表中只写"续住" |
| 费用列 | 每行记录金额，末尾汇总本日费用 |
| 火车信息 | **必须查12306获取**，禁止留"待补充"等占位符 |
| 大交通格式 | 示例：D2008（成都东07:50→剑门关09:21），二等座¥104 |
| 每日安排 | 包含从出发到入睡的完整时间线，节点紧凑 |
| 海外费用 | **所有费用统一换算为人民币（¥）**估算，原币种金额在备注标注（如 `¥350 / $50`）。汇率使用当前实时汇率，标注换算日期 |
| 多日行程详情 | **多日行程需为每个重要活动日扩展独立的行程详情**（文化分类、景点介绍、诗词引用等），避免只给一个笼统的目的地概述。每日常规移动日可简并 |

## 六、行程详情内容规范

### 6.1 信息分级质量标准

| 类别 | 要求 | 示例 |
|------|------|------|
| **事实类** | 简洁、清晰、正确、可验证 | D2008 成都东07:50→剑门关09:21，二等座¥104；小娅子→断碑梁 12.7km（GPS实测） |
| **提示类** | 保证有效性和正确性，必须验证 | ⚠️ 石洞沟、抄手铺有补给（实测有商店）；⚠️ 全程无商店需自带干粮（必须实地确认） |
| **介绍类** | 有趣、丰富、有学习价值，引用有据可查 | 古蜀道历史、金牛道典故、诗词原文+背景、美食文化 |

> ⚠️ **提示类常见错误**：
> - 说"全程无商店"但沿途实际有补给点 → 必须实地确认或查看真实攻略
> - 说"路况良好"但实际是野路 → 查看近期真实用户评论
> - 说"没有野生动物"但当地确实有 → 查阅官方信息

### 6.2 目的地详情类别（按需选用）

根据目的地特性选择 3-5 个最相关的类别，按相关性排序（参考小红书/马蜂窝/百度旅游热门标签）。

| 类别 | 适用场景 | 典型目的地 |
|------|---------|---------|
| 地理风貌 | 地形、气候、植被、景观特色 | 所有目的地 |
| 历史渊源 | 重要历史节点、战争、人物 | 古道、关隘、战场遗址 |
| 文化典故 | 诗词、传说、著名典故发生地、人物故事 | 蜀道、庐山、岳飞庙 |
| 人文与诗词 | 有诗词典故的目的地 | 蜀道、庐山、长江三峡 |
| 遗存遗迹 | 路面、驿站、关隘、古建筑、古镇 | 古道、古城、古关 |
| 美食特产 | 特色美食、风味小吃 | 所有目的地 |
| 世界遗产 | 世界遗产/预备项目及核心价值 | 蜀道、故宫、莫高窟 |
| 5A级景区 | 国家5A景区核心吸引物 | 剑门关、黄山、九寨沟 |
| 宗教文化 | 著名宗教发源地、石窟、寺庙、道观 | 五台山、武当山、敦煌 |
| 动植物 | 珍稀物种、生态特色 | 卧龙、大熊猫基地 |
| 地质地貌 | 丹霞、卡斯特、沙漠、冰川等特殊地貌 | 张掖丹霞、黄龙 |
| 民俗节庆 | 当地节日、习俗、手工艺 | 少数民族地区 |

> **排序规则**：结合小红书/马蜂窝/百度旅游热门标签，按相关性高的优先展示。例如蜀道优先展示：遗存遗迹 > 历史渊源 > 人文与诗词 > 世界遗产 > 美食特产

### 6.3 徒步路线分类

| 分类 | 定义 | 示例 |
|------|------|------|
| 🚶 穿越 | 起终点不同，A点进B点出 | 大觉寺→阳台山→凤凰岭 |
| 🔄 环线 | 起点=终点，路径不重复经过已走点 | 姜庄子村→大海坨→大海陀村→姜庄子村 |
| 🔙 往返 | 起点=终点，原路返回途经已走路线 | 大觉寺→阳台山→大觉寺 |

### 6.4 诗词引用规范

- 标题：**《诗词名》** — 作者（朝代）
- 原文：使用中文引号` `` `` `包裹原文
- 背景：1-2句话说明创作时间、地点、背景
- 赏析（可选）：1句话点出与目的地的关联

### 6.5 隐私声明

> 行程规划过程中会向 Wikipedia、web_search、xiaohongshu 等第三方服务发送目的地名称及关联查询（景点、路线、美食、酒店关键词）。传输内容**不包含**出行日期、人员数量、订单信息等个人行程数据；仅发送与目的地内容检索相关的最小必要查询词。

## 七、火车票查询规范（强制）

**所有火车票查询必须使用 `12306-train-assistant` skill**

### 7.1 工具位置

```
12306-train-assistant/client.py
```

### 7.2 查询命令（无需登录）

> ⚠️ 用户指定日期 → 查询指定日期；未指定 → 查询今天+明天；返程日必须明确查询。

```bash
# 1. 余票查询（默认查明天）
python3 client.py left-ticket --date YYYY-MM-DD --from 出发站 --to 到达站

# 2. 经停站查询（验证车次是否正确）
python3 client.py route --train-code 车次号 --date YYYY-MM-DD --from 出发站 --to 到达站

# 3. 中转方案查询
python3 client.py transfer-ticket --date YYYY-MM-DD --from 出发站 --to 到达站 --limit 10
```

### 7.3 验证清单

- [ ] 车次号在 12306 可查到
- [ ] 出发/到达站匹配用户需求
- [ ] 余票充足（或已购票）
- [ ] 经停站合理（不要坐过站）
- [ ] 票价与 plan 一致

### 7.4 已知限制

- 12306 风控触发时，换日子再试，或使用 flyai 的机票/酒店功能
- flyai 的火车票数据**不如12306准确**（例：D2008直达误显示为需绵阳换乘），**禁止用flyai替代12306查火车**
- 如遇风控报错，稍后重试

## 八、路线距离校验规范（强制）

**所有城际/乡级交通段和徒步段距离，必须用 `amap-lbs-skill` 校验**。

### 8.1 工具脚本位置

```
amap-lbs-skill/scripts/route-planning.js
amap-lbs-skill/scripts/poi-search.js
```

### 8.2 步骤一：地理编码（获取各节点经纬度）

```bash
curl -s "https://restapi.amap.com/v3/geocode/geo?address=节点名称&city=城市名&output=JSON&key=***"
```

### 8.3 步骤二：驾车路线规划（城际/包车段）

```bash
node amap-lbs-skill/scripts/route-planning.js \
  --type=driving \
  --origin=经度,纬度 \
  --destination=经度,纬度
```

### 8.4 步骤三：步行路线规划（徒步段）

```bash
node amap-lbs-skill/scripts/route-planning.js \
  --type=walking \
  --origin=经度,纬度 \
  --destination=经度,纬度
```

> ⚠️ 高德步行路线走的是现代公路网，与古道/山野徒步路线不同。
> - 古道实测距离 **<** 高德步行距离（古道是直接山野路径，高德绕行乡村公路）
> - 应以 plan 中标注的古道实测距离为准，高德数据用于**时间偏差参考**

### 8.5 校验触发条件

在以下场景**必须**执行距离校验：
- 城际包车/班车时间存疑时
- 徒步路线距离无实测数据时
- 行程节点衔接时间紧张时（需要确认是否可行）
- 用户提出"这个时间够不够"时
